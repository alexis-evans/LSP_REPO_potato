In comparing my Assignment 2 and Assignment 3 implementations of the ETL pipeline, the biggest shift is that Assignment 2 was primarily procedural while Assignment 3 is organized around objects that represent the problem domain and the major responsibilities of the system. In Assignment 2, the pipeline steps felt like a single “flow” of logic that processed raw CSV lines and carried intermediate values through a sequence of operations. In Assignment 3, I decomposed the same behavior into classes that each capture one clear abstraction: Product as the domain model, PriceRange as a classification concept, ProductCsvCodec as the CSV boundary for parsing and writing, ProductTransformer as the business rules layer, and ETLPipeline as the orchestrator that coordinates extract → transform → load. This decomposition aligns directly with the design heuristics from class: each class captures one key abstraction (Heuristic 2.8), non-related behavior is spun off rather than being bundled into one place (Heuristic 2.10), and the system’s work is distributed horizontally instead of creating a single “god” component (Heuristics 3.1 and 3.2). I also deliberately reduced class proliferation by consolidating CSV parsing and writing into a single codec and by keeping the run statistics as an inner immutable Result type inside ETLPipeline, rather than a standalone “data accessor” class, which kept the design focused without expanding the public surface area unnecessarily.

Assignment 3 is more object-oriented because it forces me to treat pipeline data as structured objects with well-defined responsibilities, rather than as loosely managed strings and primitives. The clearest OO ideas I used were classes/objects and encapsulation. Product hides its state with private final fields and supports change through “with” methods that return new objects, which keeps related data and behavior together (Heuristic 2.9) and prevents external code from mutating internal state (Heuristic 2.1). Similarly, ProductCsvCodec centralizes CSV knowledge (delimiter assumptions, validation rules, header format, and output formatting), so the rest of the system depends only on its public interface rather than re-implementing parsing/writing logic in multiple places (Heuristic 2.2 and Heuristic 2.7). I kept each class’s public protocol small and intentional—ETLPipeline exposes the pipeline run operation and returns a Result, the codec exposes parse/writeHeader/writeProduct, and the transformer exposes one transformation entry point—so the public interface stays minimal and does not leak implementation details (Heuristics 2.3, 2.4, 2.5, and 2.6) . For this assignment’s scope, I did not meaningfully use inheritance or polymorphism, because the requirements are a single fixed pipeline with one transformation strategy; forcing inheritance here would be artificial rather than improving clarity. Instead, I focused on achieving clean composition and low coupling through collaboration between small classes.

A key concern I addressed from earlier iterations was ensuring the output file behavior is correct even when the input exists but contains no data. In Assignment 2, the header logic could drift toward “whatever the input header was plus PriceRange,” which breaks down when the file is empty or otherwise malformed. In Assignment 3, the output header is always the required header written by the codec up front, independent of input contents, and the pipeline only proceeds to parse/transform for subsequent non-header lines . I also tightened edge-case behavior and reporting by returning an explicit Result object from run() and printing a consistent run summary (non-header rows encountered, transformed rows, skipped rows, and output path), while still handling missing input files gracefully without a stack trace.

To confirm Assignment 3 works the same as Assignment 2, I stress tested the new pipeline using the exact same test approach I used previously. I ran the robust sample input to validate normal-case behavior and ensure the transformed output rows match expectations (uppercased names, electronics discount, category reclassification when appropriate, and price range classification based on the final rounded price). I also tested the edge cases that historically break procedural pipelines: an empty file, an empty file with only a header, no input file at all, and several custom test cases designed to exercise row skipping rules (blank lines, wrong column counts, non-numeric IDs/prices, and empty required fields). For each test, I compared both the output CSV and the run summary counts against my Assignment 2 results to verify that the external behavior stayed consistent even though the internal structure changed significantly. Overall, Assignment 3 produces the same functional output as Assignment 2, but the OO version is easier to reason about, easier to extend, and more aligned with the heuristics we covered in class because responsibilities are separated cleanly, data is hidden behind stable interfaces, and the design avoids centralized “do-everything” logic.
