In comparing my Assignment 2 and Assignment 3 implementations of the pipeline, the most significant difference is the shift from a procedural, step-by-step design to a fully object-oriented architecture that aligns with the design heuristics we discussed in class. In Assignment 2, the pipeline logic was centralized and sequential. The extraction, transformation, and loading steps were implemented as a flow of functions operating on raw data structures. In contrast, Assignment 3 decomposes the system into focused classes such as `ETLPipeline`, `ProductCsvCodec`, `ProductTransformer`, `Product`, and `PriceRange`. Each class captures a specific abstraction and collaborates with the others through clearly defined interfaces.

Assignment 3 is more object-oriented because it models real-world concepts explicitly. Instead of passing around strings or loosely structured data, I created a `Product` class that represents a product as an object with state and behavior. Similarly, `PriceRange` models a domain concept rather than embedding conditional price logic directly inside procedural code. This reflects Heuristic 3.6, which encourages modeling the real world whenever possible. The design avoids turning simple operations into classes (Heuristic 3.9), since each class represents a meaningful abstraction rather than just a single action. For example, `ProductTransformer` encapsulates transformation rules related to products, not just a one-off function.

The new design strongly satisfies Heuristic 2.1: all data is hidden within its class. Fields inside `Product` and `PriceRange` are not manipulated directly by other classes. Instead, interaction occurs through public methods. This also supports Heuristic 2.2, since users of a class depend only on its public interface. For example, `ETLPipeline` depends on the public methods of `ProductCsvCodec` and `ProductTransformer`, but those classes are not dependent on `ETLPipeline`. This maintains proper directional dependency.

The public interfaces are minimal and intentional. The classes expose only the behavior necessary for collaboration, which satisfies Heuristics 2.3, 2.4, 2.5, and 2.6. There are no unnecessary helper methods exposed publicly; implementation details remain private. The protocol of each class is small and focused. This also avoids excessive accessor methods, addressing Heuristic 3.3. Rather than exposing raw data everywhere through getters and setters, behavior is kept close to the data it operates on (Heuristic 2.9).

The design also demonstrates Heuristic 2.7: classes interact only through public interfaces and do not rely on the internal details of other classes. Coupling is limited to export coupling. For example, the pipeline coordinates the process but does not reach inside other classes to manipulate their internal state. Each class captures one and only one key abstraction (Heuristic 2.8). `Product` represents a product. `PriceRange` represents price categorization. `ProductCsvCodec` handles CSV encoding and decoding. `ProductTransformer` handles transformation rules. Non-related behavior has been spun off into separate classes (Heuristic 2.10), preventing unrelated logic from being grouped together.

Compared to Assignment 2, the intelligence of the system is distributed more horizontally (Heuristic 3.1). In the procedural design, much of the logic was centralized, which risked creating a “god class.” In Assignment 3, the work is shared more uniformly across classes. Although `ETLPipeline` coordinates the workflow, it does not perform all logic itself, helping to avoid violating Heuristic 3.2. Responsibilities are divided so that no single class dominates the system.

The design also avoids non-communicating behavior within classes (Heuristic 3.4). Each method inside a class contributes directly to its core responsibility. There are no irrelevant classes (Heuristic 3.7) or classes outside the system’s concern (Heuristic 3.8). Each abstraction directly supports the pipeline’s purpose. Additionally, the abstractions modeled are true classes rather than simply temporary roles objects play (Heuristic 2.11), because each class represents a stable concept within the domain.

To confirm that Assignment 3 works the same as Assignment 2, I tested the new pipeline using the exact same stress tests as before. I ran it with the robust sample input to confirm correct normal behavior. I also tested edge cases, including an empty file, an empty file containing only a header, no input file at all, and several custom test cases I designed myself. In each case, I compared the behavior and output to the results from Assignment 2. The outputs matched, and error handling behaved consistently. This confirmed that while the internal design changed significantly, the external behavior of the system remained functionally equivalent.

In summary, Assignment 3 represents a clear improvement in design quality. It adheres closely to the object-oriented heuristics discussed in class by hiding data, minimizing interfaces, distributing responsibilities, avoiding god classes, and modeling real-world abstractions. While both assignments achieve the same functional result, Assignment 3 demonstrates a more maintainable, modular, and conceptually sound object-oriented architecture.
